# Design Decisions

Осознанные отличия SlipperOS от Linux, в пользу простоты и embedded.

## 1. Свой формат бинарников (SPX), а не ELF

**Проблема Linux**: ELF64 — сложный формат с секциями, релокациями, динамической линковкой, GOT/PLT, .interp, .dynamic, .init_array. Для embedded-загрузчика это оверхед.

**Решение SlipperOS**: формат SPX (SlipperExec) — 344 байта заголовок, 40 байт на сегмент. Никаких релокаций, GOT, динамической линковки. Что загрузил — то и выполняется.

**Польза**: загрузка в 10 раз быстрее, парсер — 100 строк C, никаких страданий с динамической линковкой. Для embedded — идеал.

## 2. Нет линуксовой модели драйверов

**Проблема Linux**: device tree, platform drivers, driver model, deferred probe, модули, power management — сложнейшая подсистема на тысячи строк.

**Решение SlipperOS**: `fdt_find_uart()` → MMIO адрес → работаем. Всё в ядре, никаких модулей. Железо определяется по FDT и инициализируется напрямую.

**Польза**: инициализация железа за микросекунды, а не секунды (как Linux на Duo S). Нет deferred probe, нет dependency hell.

## 3. Тулзы на C, а не на Python

**Проблема Linux**: buildroot, Yocto, genimage, mkimage — на Python. Для сборки ядра нужно тащить интерпретатор, пакеты, виртуальное окружение.

**Решение SlipperOS**: `elf2spx.c` (ELF → SPX), `mkimage.c` (сборка SlipperFS). 150 строк на C каждая, компилируются за 0.1 секунды, без единой зависимости.

**Польза**: бинарник собран один раз — работает где угодно. Нет `ModuleNotFoundError`, нет `pip install`, нет Python тулчейна. Make + GCC = всё, что нужно.
